## What Is an SBOM?

An SBOM, or software bill of materials, is an inventory of components in a software artifact. For a container image, it may include operating system packages, npm packages, files, versions, suppliers, licenses, and relationships.

The simplest reason to keep an SBOM is speed. When a new vulnerability or malicious package is announced, the team needs to know which artifacts included the affected component.

```text
Question: Did production run package example-parser 1.4.2?
Evidence: SBOM for image sha256:4e1b9f30...
Answer: yes, through express -> body-parser -> example-parser
```

Without an SBOM, the team may need to rebuild old images, inspect registries manually, or guess from source files that have changed since release.

## What It Records

A small SBOM excerpt might look like this:

```json
{
  "name": "devpolaris-orders-api",
  "version": "2026.05.19.1",
  "components": [
    {
      "name": "express",
      "version": "4.18.2",
      "type": "library"
    },
    {
      "name": "openssl",
      "version": "3.0.13",
      "type": "operating-system-package"
    }
  ]
}
```

The `name` and `version` fields identify the artifact. Each component records a package name, version, and type. Larger SBOMs may include package URLs, hashes, licenses, supplier names, and dependency relationships.

The SBOM should be tied to an artifact digest:

```text
Image: ghcr.io/devpolaris/orders-api@sha256:4e1b9f30...
SBOM: orders-api-sha256-4e1b9f30.spdx.json
Generated by: release workflow #1842
Stored in: release evidence bucket
```

If the SBOM is not tied to a digest, it is hard to prove which image it describes.

## SBOM Formats

The two common SBOM format families are SPDX and CycloneDX. Both can represent component inventories. The right choice often depends on the tools your scanner, registry, and compliance process already support.

| Format | Common use | What to check |
|--------|------------|---------------|
| SPDX | Licensing, package inventory, compliance exchange | Tool support and package identifiers |
| CycloneDX | Security and component analysis | Vulnerability and dependency relationship support |

The beginner habit is simple: pick one supported format, generate it automatically during release, and store it beside the artifact digest.

## Using SBOMs During Response

Suppose a package advisory arrives.

```text
Advisory: example-parser 1.4.2 malicious
Question: which released images included it?
Search target: SBOM component name=example-parser version=1.4.2
Result: orders-api sha256:1111..., orders-worker sha256:3333...
Action: rebuild and redeploy both images, rotate secrets from affected CI jobs
```

The SBOM turns a broad question into a lookup. It also helps avoid unnecessary work. If a service never shipped the component, the team can record that evidence and move on.

SBOMs are also useful for audit evidence. They show that the release process captured the component inventory at the time of build. That is stronger than trying to recreate the inventory months later.

## SBOM Limits

An SBOM is an inventory, not a guarantee of safety. It may miss dynamically downloaded code. It may be wrong if the generator cannot read a package format. It does not prove the build was trustworthy. It does not say whether a vulnerable function is reachable.

Use SBOMs with other evidence:

```text
SBOM -> what components shipped
Scanner -> what known risks match those components
Provenance -> how the artifact was built
Signature -> which identity endorsed the artifact
Deployment record -> where the artifact ran
```

The SBOM answers "what is inside?" The other records answer how it was built, who approved it, and where it ran.

## Putting It All Together

An SBOM is the component inventory for an artifact. For `devpolaris-orders-api`, the useful SBOM is generated during the release workflow, tied to the image digest, stored with release evidence, and searchable during incident response.

SBOMs are most useful when a new event happens after release: a vulnerability, malicious package report, license question, or customer evidence request. The team can search what shipped instead of guessing from today's source tree.
